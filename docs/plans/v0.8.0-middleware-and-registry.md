# v0.8.0 ŌĆö Tool registry, middleware chain, and composable extensions

**Status**: Draft
**Target version**: v0.8.0 (additive; no breaking changes)
**Split**: PR A (#1ŌĆō#5, the foundation + cheap features), PR B (#6ŌĆō#9, heavier self-contained features)

---

## 1. Context & motivation

Today, consumers of `mcpserver` implement a single `ToolHandler` with a large
`switch` inside `Call()` covering every tool. There is no seam between the
transport and the tool handler for cross-cutting concerns ŌĆö response shaping,
logging, input validation, suggestion on typo, audit trail, elevation gating ŌĆö
so every consumer either reimplements these inline or does without.

The Node.js `atlassian-mcp-server` has spent two years accreting the missing
layer: command builder + auto-discovery, mux dispatcher (191 tools ŌåÆ 4),
compact-response transformer, fuzzy-suggestion error path, diagnostic SQLite
ring-buffer logger, config-KV seeding with write-through, webhook router with
HMAC verification. Every one of those pieces is reusable in any non-trivial
MCP server. They only exist in app code because there was no library seam to
hang them on.

This plan adds that seam and lands the generic features on top of it. The
immediate consumer is the forthcoming Go port of `atlassian-mcp-server`, but
the target shape is "useful to any MCP server with >50 tools or any
non-trivial REST-wrapping logic."

## 2. Goals & non-goals

### Goals
- Ship a **tool registry** that lets tools register themselves with metadata
  instead of one monolithic `ToolHandler`.
- Ship a **tool-call middleware chain** as the composition seam for everything
  below.
- Ship middleware/utility features that use the seam: response transformer,
  mux dispatcher, fuzzy suggester, parameter aliasing, schema validator,
  diagnostic logger.
- Ship two observability hooks that pair with the suggester and aliaser:
  unknown-name suggestion telemetry (callback + JSONL default) and a
  registry-driven skill-content builder for the mux's `_get_skill` tool.
- Ship two adjacent helpers that consumers repeatedly reinvent: config KV
  store with seeding precedence, webhook router with HMAC verification.
- Zero breaking changes. All features opt-in via functional options.

### Non-goals
- **No MCP client subpackage.** That is a separate conversation (v0.9.0).
- **No resources / prompts protocol support.** Deliberate; tools-only is the
  library's current shape and we're not changing it here.
- **No `log/slog` migration of the existing codebase.** The diagnostic logger
  (#7) uses `slog` internally for its own purposes; the rest of the library
  stays on stdlib `log`.
- **No changes to the auth module.** OAuth, elevation, rate limiting, SQLite
  auth store ŌĆö all untouched.
- **No metrics / Prometheus integration.** Middleware chain makes this a
  one-option addition later; we don't need to ship it now.
- **No runtime tool registration.** Registry is built at startup; registration
  after `ListenAndServe` is an error.

## 3. Design decisions

Each is framed as **Decision ŌåÆ Rationale ŌåÆ Alternatives considered**. Locking
these here prevents mid-PR litigation.

### 3.1 Registry is an implementation of `ToolHandler`, not a replacement

**Decision**: Add a new `Registry` type with `Register(Tool)` and
`AsToolHandler() ToolHandler` methods. The existing `ToolHandler` interface
stays unchanged.

**Rationale**: Backward compatibility. Every existing consumer keeps working
without changes. New consumers get the ergonomic API; power users who need
custom dispatch keep the raw interface.

**Alternatives considered**:
- Replace `ToolHandler` with `Registry` ŌåÆ breaks every existing consumer.
- Add `Registry` alongside `ToolHandler` as independent surface ŌåÆ two ways to
  do the same thing with no clear "which one do I use?" guidance.

### 3.2 Middleware signature: function wrapper, not interface

**Decision**:
```go
type ToolCallFunc func(ctx context.Context, name string, args map[string]any) (result any, isError bool, err error)
type ToolMiddleware func(next ToolCallFunc) ToolCallFunc
```

**Rationale**: Matches stdlib `net/http` middleware idiom. Composable without
boilerplate (closures). Interface-based middleware forces each middleware
into its own struct + method, which is noisy for tiny transforms.

**Alternatives considered**: Interface with `Before(ctx) / After(ctx, result)`
hooks. Rejected ŌĆö it can't express short-circuit behavior (e.g. fuzzy
suggester returning early on unknown tool name) without awkward sentinel
returns.

### 3.3 Error return becomes first-class

**Decision**: `ToolCallFunc` returns `(result any, isError bool, err error)`.
`err != nil` means "library/middleware-level failure" (schema validation
failed, middleware panicked). `isError: true` with `err == nil` means
"tool ran successfully and is reporting a tool-level error" (404 from a REST
call, etc.) ŌĆö matches MCP protocol semantics.

**Rationale**: Today the library's `ToolHandler.Call` returns `(any, bool)`,
conflating these two error categories. Middleware needs to distinguish
them (e.g. schema-validator middleware should short-circuit with `err`; a
404 from a remote service is a successful tool call reporting failure).

**Alternatives considered**: Keep `(any, bool)` and encode library errors as
`isError: true` with a magic prefix. Rejected ŌĆö brittle and leaks
implementation details to the wire.

**Backward compat**: The existing `ToolHandler` interface is kept with its
current `(any, bool)` signature. The registry's internal `ToolCallFunc`
adapts by treating any `error` return from a registered tool as `(nil, true,
err)` and formatting the error text into the MCP content array. Consumers
using the raw `ToolHandler` interface see no change.

### 3.4 Schema validation dep: `santhosh-tekuri/jsonschema/v6`

**Decision**: Use `github.com/santhosh-tekuri/jsonschema/v6` for JSON Schema
input validation.

**Rationale**: Pure Go, draft 2020-12 support, well-maintained, minimal
transitive deps (one: `golang.org/x/text` for idna). The alternatives either
lack draft 2020-12 or pull in large trees.

**Alternatives considered**:
- Roll our own minimal validator for the subset of JSON Schema the Atlassian
  tools use ŌåÆ trap. We'd re-discover every edge case of draft 2020-12 over
  the next 18 months.
- `xeipuuv/gojsonschema` ŌåÆ draft 7 only, unmaintained.
- Skip validation; keep it app-level ŌåÆ we already tried that; it's why every
  consumer reinvents the wheel.

### 3.5 Diagnostic logger storage: separate SQLite file

**Decision**: Diagnostic logs live in their own SQLite file (default
`/data/server-logs.db`), not the OAuth DB.

**Rationale**: Log volume is orders of magnitude higher than auth writes.
Mixing them risks contention on the single-writer connection the auth
module deliberately enforces, and a log-truncation bug could corrupt auth
state. Separate file also means log DB can be deleted for privacy without
wiping OAuth clients.

**Alternatives considered**: Shared DB with WAL handling contention.
Rejected ŌĆö auth module's single-writer pool is a load-bearing design
decision; we don't want to revisit it.

### 3.6 Mux tool shape: 4 tools, configurable names, ship defaults

**Decision**: Mux exposes exactly 4 tools: `<prefix>_execute`,
`<prefix>_list_tools`, `<prefix>_get_skill`, `<prefix>_health`. Prefix
configurable; tool set not configurable in v0.8.0.

**Rationale**: The 4-tool shape is battle-tested in the Node server and
several sibling MCP servers. Making it configurable in v1 invites
bike-shedding and increases the surface area we have to test.

**Alternatives considered**: Let consumers pass their own mux tool
definitions. Deferred to v0.9.x if anyone actually needs it.

### 3.7 Fuzzy matcher: Levenshtein, in-tree, no dep

**Decision**: Implement Levenshtein distance and "top-N closest" inline as an
unexported helper under `suggest/`. No external dep.

**Rationale**: ~60 LOC, trivial to test, avoids pulling a fuzzy-matching
library for what is essentially a textbook algorithm.

**Alternatives considered**: `github.com/sahilm/fuzzy` ŌĆö pulls `mattn/go-runewidth` transitively. Overkill for the use case.

### 3.8 Config KV store: SQLite + file write-through + env fallback

**Decision**: `config.Store` reads in order SQLite ŌåÆ file ŌåÆ env; writes go to
both SQLite and file. File path configurable; default follows the Atlassian
convention of `/opt/CREDENTIALS.txt` for deployment resilience but the path
is not imposed on consumers.

**Rationale**: The precedence chain is the whole point. Apps that only want
env-var config can skip this feature entirely.

**Alternatives considered**: HashiCorp Vault / Consul integration. Out of
scope; explicit non-goal of "minimal deps."

### 3.9 Parameter aliases as their own middleware, not a validator quirk

**Decision**: Aliasing lives in a dedicated `WithParamAliases(global)` option
plus a `Tool.ParamAliases map[string]string` field. Resolution runs as its own
middleware that precedes schema validation in the chain.

**Rationale**: LLMs consistently guess short parameter names (`issueKey`,
`key`, `projectKey`) instead of canonical ones. In the Node server the single
`GLOBAL_PARAM_ALIASES` map covers eight aliases and is the largest source of
"did you mean?" hits. Folding alias resolution into the schema-validator
middleware would tie it to validation being installed ŌĆö but aliases want to
work when a consumer opts out of validation (dev mode, or while drafting
schemas). A standalone middleware keeps composition independent.

**Resolution rule** (same as Node `resolveParamAliases`): rewrite
`alias ŌåÆ canonical` iff `alias Ōłł args Ōł¦ canonical Ōłł tool.schema.properties
Ōł¦ canonical Ōłē args`. Per-tool `Tool.ParamAliases` override the global map.

**Alternatives considered**: Bake aliases into the validator. Rejected ŌĆö
forces a PR B feature to enable a PR A feature, violating the composition
goal.

**Default aliases shipped**: none. The library's global alias map is empty
by default; consumers declare their own via `WithParamAliases`. The
Atlassian Go port will declare its own set (mirroring Node's
`GLOBAL_PARAM_ALIASES`) in app code, not in the library. Rationale: alias
sets are domain-specific, and shipping a curated list risks a long tail of
"why is *this* alias built-in?" churn.

### 3.10 Unknown-name telemetry via callback, with JSONL default

**Decision**: `WithSuggestionHook(func(suggest.Event))` option. The fuzzy
suggester (used by Mux dispatch and, when installed, the validator's
unknown-property path) fires the hook once per unknown name, with a
`Kind` ("tool"/"param"), `Requested`, `Suggested`, parent `Tool`, and
`Timestamp`. A prebuilt `suggest.JSONLFile(path)` hook covers the common
case (appends one JSON object per line, fire-and-forget, file-system errors
dropped).

**Rationale**: Mining this telemetry is how you grow the alias map over
time ŌĆö it is the feedback loop that makes ┬¦3.9 pay off. But imposing a file
path on every consumer breaks read-only container deployments and duplicates
work for consumers that already have structured logs. A callback is ~10 LOC
and defers the format choice; shipping the JSONL hook means the lazy path
still works with one option line.

**Alternatives considered**: Hardcode JSONL output at a configurable path.
Rejected for the read-only-container reason above. Route through the diag
logger instead. Rejected ŌĆö diag is PR B and is a different category
(ops-facing audit vs. product-facing signal).

### 3.11 Skill content is built from the registry, not hand-written

**Decision**: `MuxConfig.Skill` is a `SkillBuilder func(r *Registry) string`
(supersedes the `SkillContent func() string` field from the earlier sketch).
A default `DefaultSkillBuilder(opts)` ships that generates: header and
how-to snippet, curated top-N with parameter signatures, categorised listing
grouped by `Tool.Category`, and a footnote pointing at `<prefix>_list_tools`.

**Rationale**: The Node server's `buildSkillContent` derives ~90% of its
output from the registry; only the curated top-20 sections and narrative
prose are hand-written. Every consumer who stands up a mux will otherwise
reinvent this loop. Ship the default renderer and let consumers override
just the narrative via `SkillOptions.Intro`.

**Parameter-signature source**: `DefaultSkillBuilder` derives signatures by
parsing the top level of `Tool.InputSchema` at render time. It reads
`properties` (names + `type` + `description`) and `required` (names array).
The parse handles only the flat top-level shape ŌĆö if a tool's schema uses
`$ref`, `oneOf`, or `allOf` at the root, the parser gives up gracefully and
renders the tool as `toolName(...)` with no arg list. Hand-written schemas
that are flat objects (the overwhelming majority in practice) render full
signatures.

**Rationale for parse-over-duplicate**: A parallel `Tool.Params` display
field is a second source of truth ŌĆö in the Node server the equivalent
metadata field and the JSON schema visibly drift over time. Parsing the
schema once per skill render (milliseconds, cached with the skill output)
trades a small amount of complexity for single-source-of-truth guarantees
forever.

**Alternatives considered**:
- Parallel `Tool.Params []ParamDoc` display field. Rejected for the
  drift reason above; easier to get right initially but rots over time.
- Require consumers to provide their own renderer. Rejected ŌĆö it is the
  single most copy-pasted piece of Node boilerplate in the portfolio, and
  every copy has diverged.

## 4. Public API sketches

Package-level additions. All under the root `mcpserver` package unless noted.

### 4.1 Registry (`registry.go`)

```go
// Tool is a self-describing unit of functionality registered with a Registry.
type Tool struct {
    Name         string
    Description  string
    InputSchema  json.RawMessage    // opaque; validated by schema middleware if installed;
                                    // DefaultSkillBuilder also reads top-level properties/required for signature rendering
    Category     string             // optional; used by Mux for grouping
    Tags         []string           // optional; freeform labels (e.g. "readOnly", "destructive")
    ParamAliases map[string]string  // optional; per-tool alias ŌåÆ canonical rewrites
    Handler      func(ctx context.Context, args map[string]any) (any, error)
}

// Registry is an ergonomic alternative to implementing ToolHandler directly.
// A Registry implements ToolHandler; pass the result of AsToolHandler to
// Server.RegisterTools.
type Registry struct { /* unexported */ }

func NewRegistry() *Registry
func (r *Registry) Register(t Tool) error       // duplicate name ŌåÆ error
func (r *Registry) MustRegister(t Tool)         // panic variant for init()
func (r *Registry) Lookup(name string) (Tool, bool)
func (r *Registry) All() []Tool                 // stable-sorted by Name
func (r *Registry) Categories() []string
func (r *Registry) AsToolHandler() ToolHandler  // satisfies existing interface
```

### 4.2 Tool-call middleware (`middleware.go`)

```go
// ToolCallFunc is the signature every tool and middleware produces.
// Returning err != nil indicates a library-level failure (validation,
// middleware panic). isError: true with err == nil indicates a successful
// tool call that is reporting a tool-level error to the client.
type ToolCallFunc func(ctx context.Context, name string, args map[string]any) (result any, isError bool, err error)

// ToolMiddleware wraps a ToolCallFunc with additional behaviour.
type ToolMiddleware func(next ToolCallFunc) ToolCallFunc

// WithToolMiddleware installs one or more middlewares. They run in the
// order passed (outermost first); the innermost wraps the registry's
// dispatch. Safe to call multiple times; middlewares accumulate.
func WithToolMiddleware(mw ...ToolMiddleware) Option
```

### 4.3 Response transformer (`transformer.go`)

```go
// ResponseTransformer mutates or replaces the result of a successful tool
// call before it is serialised to the wire. Transformers are installed as
// middleware and run after the tool returns.
type ResponseTransformer func(toolName string, result any) any

func WithResponseTransformer(t ResponseTransformer) Option

// CompactResponse returns a transformer that strips the given JSON field
// names (at any depth) and optionally drops nil/empty values. Ships with
// sensible REST-envelope defaults via CompactREST().
func CompactResponse(opts ...CompactOption) ResponseTransformer
func StripFields(fieldNames ...string) CompactOption
func StripNulls() CompactOption
func StripEmptyArrays() CompactOption
func CompactREST() ResponseTransformer // preset: self, expand, avatarUrls, iconUrl, _links, nulls
```

### 4.4 Mux (`mux.go`)

```go
type MuxConfig struct {
    Prefix             string                                              // e.g. "atlassian" ŌåÆ atlassian_execute
    Skill              SkillBuilder                                        // renders <prefix>_get_skill content; nil ŌåÆ DefaultSkillBuilder(SkillOptions{})
    HealthCheck        func(ctx context.Context) (map[string]any, error)
    Compact            bool                                                // default true; when true, execute applies CompactREST
    GlobalParamAliases map[string]string                                   // optional; applied to every dispatched tool (┬¦4.10)
}

// NewMux wraps a Registry and returns a new Registry that exposes exactly
// 4 dispatcher tools instead of the underlying tools. The original tools
// are dispatched by <prefix>_execute with {tool, args}.
func NewMux(underlying *Registry, cfg MuxConfig) *Registry

// SkillBuilder renders the markdown content returned by <prefix>_get_skill.
// It receives the underlying registry (not the mux's 4-tool shell) so it
// can list the real tools.
type SkillBuilder func(r *Registry) string

// DefaultSkillBuilder returns a SkillBuilder that renders:
//   1. Header + Intro (from opts)
//   2. Quick-reference sections built from opts.Featured (with parameter
//      signatures derived from each tool's top-level InputSchema properties
//      and required array; tools with non-flat root schemas render as
//      `toolName(...)`)
//   3. "All operations by category" grouped by Tool.Category
//   4. Footnote pointing at <prefix>_list_tools for full schemas
// Featured tool names that are not in the registry are skipped (logged via
// stdlib log ŌĆö diag may not be installed).
func DefaultSkillBuilder(opts SkillOptions) SkillBuilder

type SkillOptions struct {
    Title      string     // e.g. "Atlassian MCP ŌĆö Routing Guide"; default "<Prefix> MCP ŌĆö Routing Guide"
    Intro      string     // free-form markdown prepended under the header
    Featured   []Featured // optional curated highlights
    Categorise bool       // default true; emit "## All Operations by Category"
}

type Featured struct {
    Section string   // e.g. "Jira ŌĆö Issues & Search"
    Tools   []string // ordered list of Tool.Name entries
}
```

### 4.5 Fuzzy suggester (`suggest/suggest.go`, new subpackage)

```go
package suggest

// Closest returns up to n candidates closest to query by Levenshtein

// ClosestWithScores is like Closest but returns (name, distance) pairs.
```

### 4.9 Webhook router (`webhook/webhook.go`, new subpackage)

```go
package webhook

type Verifier func(r *http.Request, body []byte) error

func HMACSHA256(secret string, headerName string) Verifier

type Router struct { /* unexported */ }

```

Verifiers return `nil` on success; non-nil error causes a 401 before the
handler runs. Bodies are buffered once (capped at 10 MB) so verifier and
handler see the same bytes.

### 4.10 Parameter aliases (`aliases.go`)

```go
// WithParamAliases installs middleware that rewrites parameter names in the
// args map before reaching the handler or any downstream middleware.
//
// Rewrite rule: for each (alias ŌåÆ canonical) in the merged map, rewrite
// args[alias] ŌåÆ args[canonical] iff
//   alias is present in args, AND
//   canonical is a declared property of the tool's Params/InputSchema, AND
//   canonical is not already present in args.
//
// Per-tool aliases (Tool.ParamAliases) override the global map on conflict.
// The alias middleware is ordered before schema validation so validators
// only ever see canonical names.
func WithParamAliases(global map[string]string) Option
```

Behaviour notes:
- A tool with no `InputSchema` has no canonical names, so aliasing is a
  no-op for it. (Canonical-name lookup parses the top-level `properties`
  key of the schema ŌĆö the same parse the skill builder uses.)
- Alias collision (alias and canonical both present in payload) preserves
  canonical and drops alias ŌĆö LLMs that get it right the second time
  shouldn't have their correction overridden. The drop fires a
  `suggest.EventAliasCollision` event so consumers can mine for LLMs that
  are hedging (sending both). The event's `Requested` is the dropped alias,
  `Suggested` is the canonical that won.

### 4.11 Suggestion telemetry (`suggest/hook.go`)

```go
package suggest

type EventKind string

const (
    EventUnknownTool    EventKind = "tool"             // Mux dispatch got a tool name not in the registry
    EventUnknownParam   EventKind = "param"            // Validator or alias middleware got a property not in the schema
    EventAliasCollision EventKind = "alias_collision"  // Alias middleware saw both alias and canonical in the payload
)

// Event is fired by the fuzzy suggester (unknown-name cases) and the alias
// middleware (collision case). For EventUnknownTool, Tool is empty. For
// EventUnknownParam and EventAliasCollision, Tool is the parent tool name.
// For EventAliasCollision, Requested is the dropped alias and Suggested is
// the canonical that won ŌĆö Suggested is never empty in this case.
type Event struct {
    Kind      EventKind
    Tool      string
    Requested string
    Suggested string // may be empty for unknown-name events when no candidate passed the threshold
    Timestamp time.Time
}

// Hook consumes suggestion events. Must be safe for concurrent use; must
// not block (the caller is on the request path).
type Hook func(Event)

// JSONLFile returns a Hook that appends each event as a single JSON line
// to path. The file and its parent directory are created on first write.
// All file-system errors are silently dropped ŌĆö telemetry never breaks
// request flow. Writes are serialised through an internal mutex so
// concurrent calls do not interleave.
func JSONLFile(path string) Hook

// Multi returns a Hook that fans out to every h in hooks in order. A hook
// that panics is recovered; subsequent hooks still run.
func Multi(hooks ...Hook) Hook
```

Server option:

```go
// WithSuggestionHook wires a suggest.Hook so that every unknown tool name
// (Mux dispatch) and every unknown parameter name (validator or alias
// middleware) produces a suggestion event after fuzzy lookup.
func WithSuggestionHook(h suggest.Hook) Option
```

Key invariants:
- The middleware chain only runs for `tools/call`. Other methods (`initialize`, `tools/list`, `ping`) bypass it ŌĆö they have no user-supplied args to transform.
- Middleware order is **outermost first**. `WithToolMiddleware(a, b, c)` means `a(b(c(dispatch)))`. Matches `net/http` convention.
- The registry exposes `ToolHandler` but middleware chains wrap the `ToolCallFunc` that ultimately calls the registry. Consumers who bring their own `ToolHandler` still get middleware because the wrapping happens at the transport layer, not inside the registry.

For each feature, minimum required tests:
1. **Happy path** ŌĆö the advertised use case works.
2. **Error path** ŌĆö each documented error mode returns the documented error.
3. **Feature-specific edge case** ŌĆö at least one "this is the bug you'd hit in production" test. Examples:
   - Mux: unknown tool name ŌåÆ fuzzy suggestion in error.
   - Transformer: nested arrays and maps, not just top-level.
   - Validator: additionalProperties under strict mode.
   - Param aliases: alias and canonical both present ŌåÆ canonical wins, alias dropped, `EventAliasCollision` fires with Requested=alias, Suggested=canonical.
   - Param aliases: per-tool alias overrides global with different canonical.
   - Suggestion hook: concurrent JSONL appends do not interleave; hook panic is isolated; parent dir auto-created.
   - Skill builder: Featured tool name absent from registry is skipped (not fatal); category order stable; tool with `oneOf` / `$ref` at schema root renders as `toolName(...)` rather than panicking; tool with flat schema but no `required` array renders all params as optional.
   - Diag logger: ring buffer eviction when exceeding RingSize.
   - Config store: SQLite write fails but file write succeeds ŌĆö what's the resolved value?
   - Webhook router: HMAC signature computed on the raw body, not on re-serialised JSON.
4. **Composition test** ŌĆö one test per PR that wires multiple features together and verifies they compose correctly. This is the integration guarantee.

No new CI infrastructure. `go test ./...` continues to cover everything.

### 8.3 Cross-session coordination rules

- **One track = one branch = one PR.** No track modifies another
  track's files. If you find you need to, stop and open a Phase 0
  amendment PR (single-agent, short-lived) first.
- **`options.go` is frozen after Phase 0.** Every `WithXxx` option is
  declared there; tracks implement the plumbing the option wires into,
  not the option signature itself.
- **`transport.go` is owned by Session 2 (track 1B).** No other session
  touches it.
- **Merge order is free after Phase 0.** Because Phase 0 is
  minimal-working, master compiles and `go test ./...` stays green at
  every commit regardless of merge interleaving.
- **Tests at track boundary.** Each track tests its public API in
  isolation. Cross-session integration lives only in Session 4's
  `composition_test.go`.
- **Sessions do not poach tracks from each other.** Track ownership is
  fixed at kickoff. A session that finishes early helps another session
  by reviewing PRs or supplying sub-agent capacity, not by adopting
  tracks.
- **Cross-session dependency mid-development = wrong allocation.** If
  you discover one, stop and escalate to the plan owner rather than
  serialising.
- **Re-read ┬¦3 design decisions before starting.** ┬¦3.9 (alias
  middleware independence), ┬¦3.10 (callback-based telemetry, alias
  collisions fire events), ┬¦3.11 (skill from schema parse, no parallel
  display field), and ┬¦3.3 (error-category contract) are the
  non-obvious ones.
- **Branch naming**: `v0.8.0/{scope}-{name}` where `{scope}` is either
  `phase-0-scaffold` for the Phase 0 prep PR, or `s{N}-{track}` for a
  session track. Track codes match the plan (1a, 1b, 2a, 2b, 2c, 2d,
  2e, 2f, 3a, 3b, 4a, 4b) in lowercase. Examples:
  - `v0.8.0/phase-0-scaffold`
  - `v0.8.0/s1-1a-registry`
  - `v0.8.0/s2-3b-mux`
  - `v0.8.0/s3-2a-suggest`
  - `v0.8.0/s4-4b-composition`
  Running `gh pr list` then clusters PRs by version and by session.
- **Review flow**: every track PR targets `master`. Review proceeds in
  two stages: **(1) session-lead review** ŌĆö the primary agent in the
  session reviews its sub-agents' PRs and blocks progression until the
  track's exit criteria are visibly met; **(2) final overall review** ŌĆö
  a single reviewer signs off after session-lead approval and merges.
  No self-merge by sub-agents; no session-lead-only merges. This is the
  only cross-session synchronisation point.

## 9. Risks & open questions

Unknowns that warrant a check before or during implementation:

1. **Does `transport.go`'s current `tools/call` handling cleanly accept a
   middleware chain?** Need to verify the handler signature and whether the
   existing call site passes `context.Context` through. If not, a small
   refactor precedes PR A.

2. **Does the existing elevation middleware interact correctly with the new
   tool-call chain?** Elevation runs at the HTTP layer today. If we want
   per-tool elevation requirements (e.g. `config_set` always requires
   elevation), we need a convention ŌĆö probably a tag like `requiresElevation`
   on `Tool`, enforced by a built-in middleware. Defer the convention to PR
   B if it's not needed for PR A.

3. **Is there a hidden global in the SQLite auth store that a separate
   diag-log DB could collide with?** Unlikely (they're separate files) but
   worth a grep for `sql.Open` side effects on init.

4. **Does `santhosh-tekuri/jsonschema/v6` actually parse the MCP-flavored
   schemas that consumers will produce?** MCP uses draft 2020-12 but tools
   often hand-write schemas that miss `$schema` declarations. Need to verify
   the library defaults to 2020-12 or provide a wrapper that sets it.

5. **Error wrapping of tool-level errors through middleware**: if the fuzzy
   suggester short-circuits with an error, does the response transformer
   still run? (Design answer: no ŌĆö err != nil skips transformer. Needs an
   explicit test.)

6. **Alias-then-validate ordering**: the plan wires alias middleware before
   schema validation so the validator only ever sees canonical names. Verify
   that when a consumer installs *only* the validator (no aliases) and *only*
   aliases (no validator), both orderings degrade gracefully. Also: confirm
   that `WithParamAliases` installed twice accumulates (matches
   `WithToolMiddleware` semantics) rather than replacing.

7. **Featured-tool dangling references in `DefaultSkillBuilder`**: tool names
   in `SkillOptions.Featured` that are absent from the registry are skipped
   with a stdlib `log.Printf` warning. Confirm the log target ŌĆö diag logger
   may not be installed, and we don't want skill-builder to hard-depend on
   it. Also: decide whether an entirely-empty Featured section (all tools
   dangling) is omitted or rendered empty.

8. **JSONL hook on read-only filesystems**: `suggest.JSONLFile` swallows
   errors, which means a misconfigured path is silent. Decide whether the
   first failed write should also call `log.Printf` once (not once per write
   ŌĆö that would defeat the point). Lean toward yes, one-shot warning.

## 11. Out of scope / follow-ups

Captured here so they are not forgotten but clearly not v0.8.0:

- **MCP client subpackage** (for calling other MCP servers). Target v0.9.0.
  Needed by the `atlassian-mcp-server` Go port's Slack + GitHub client layer.
- **Metrics / OpenTelemetry integration.** Once the middleware chain exists,
  this is a one-option addition. Target v0.8.x when a consumer actually asks.
- **Streamable HTTP transport** (MCP spec bidirectional streaming). Large
  design question of its own; not blocked by this work.
- **Runtime tool registration** (add/remove after `ListenAndServe`). Deferred
  until a concrete use case emerges.
- **Resources / prompts MCP methods.** Still an explicit non-goal.
- **Per-tenant partitioning via `client_id`** on the tool-handler signature.
  This is the LOW-severity deferred audit finding from v0.7.4; the registry
  redesign is a natural place to surface it, but we're not doing it in
  v0.8.0.
- **Phase-engine / pipeline orchestration primitives.** Too opinionated to
  upstream. Stay in app code.
